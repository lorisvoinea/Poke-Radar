## What Went Well

1. **Discipline TDD** — Toutes les stories ont suivi red-green-refactor. Story 1.2 : 11 tests UI observés en échec avant correction. Story 1.3 : 26 tests Rust + 35 tests UI + E2E Playwright.
2. **Revue de code rigoureuse** — Chaque story a eu des findings actionnables corrigés en boucle, pas de "LGTM" paresseux.
3. **Migrations non-destructives** — Story 1.3 a prouvé qu'on peut corriger un seed sans casser les données existantes (migration v4 corrective plutôt que rewrite v3).
4. **Responsive mobile-first tenu** — AC4 de la Story 1.3 validé à 320px avec mesures réelles (pas juste des assertions CSS vides).
5. **Qualité test exceptionnelle** — 35 tests UI avec mesures de viewport réelles, suite E2E Playwright complète.

## Challenges & Growth Areas

1. **Fragilité du tooling build** — Les hooks Tauri (`beforeDevCommand`, CWD, quoting Windows) ont pris 7 itérations sur Story 1.1. Résolu en sortant le build des hooks Tauri et en le mettant dans `package.json`.
2. **State management UI asynchrone** — Les stories 1.2 et 1.3 ont combattu les mêmes problèmes : feedback lié au mauvais brouillon, doubles soumissions, réponses obsolètes. Pattern `submittingRevision` découvert tardivement.
3. **Explosion de la complexité de review sur Story 1.3** — 32 findings (16 Patch + 16 Defer). 3x plus que Story 1.2. Cause : story transversale touchant migration + domaine + repository + commandes Tauri + composants UI + responsive + E2E.
4. **Absence de visibilité UI pour le Product Owner** — L'application Tauri desktop nécessite un environnement graphique. Ubuntu n'a pas pu voir l'UI réelle, seulement les maquettes et les tests.
