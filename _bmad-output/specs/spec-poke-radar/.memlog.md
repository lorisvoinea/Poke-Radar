- (decision) Web exposure: browser access to existing single-user app via domain name. Same Rust backend + React frontend, add HTTP layer. Not multi-tenant, not SaaS.
- (decision) Responsive web: mobile-first, 320px-1920px+. Mobile: 1-col cards, bottom tab bar. Desktop >=1024px: split-view Radar+Detail, side nav, dense table.
- (decision) Theme: Amber Warmth light mode (bg=#FFFDF7, accent=#D97706, text=#2D2412). Dark mode deferred post-MVP. docs/DESIGN.md still contains archived dark palette (#0B1020) — needs regeneration for light mode.
- (decision) Authentication: hors scope MVP. Access controlled at network/reverse-proxy level. Auth can be added post-MVP without breaking architecture.
- (decision) Typography: SF Pro for all UI. Sizes: titles 16px Semibold, amounts 18px Bold, labels 11px Medium uppercase, meta 12px Regular, CTA 14px Semibold, tabs 11px Medium.
- (decision) French locale: UI text in French, currency X,XX €, dates JJ/MM/AA, relative time 'il y a X min'. i18n architecture ready for future extension.
- (decision) Touch targets 44px minimum. Contrast AA minimum, AAA on KPIs. Keyboard navigation. prefers-reduced-motion respected.
- (decision) Notification channels: Telegram for critical alerts above threshold, in-app for all alerts. Push mobile reserved future (PWA/native).
- (decision) Confidence levels: ★★★ high (multi-source confirmed), ★★☆ medium (single source + historical data), ★☆☆ low (single source only), ⚠ none (manual entry).
- (decision) Degraded mode: source OK=normal, source slow=yellow badge ⏳, source down=greyed card + red badge + manual entry CTA, manual fallback=blue badge.
- (decision) Stack: Rust backend (existing) + React/Tauri frontend, adapted to web HTTP. Reverse proxy (nginx/caddy) for HTTPS termination. SQLite database.
- (decision) Core loop: Collect → Normalize → Score → Notify → Decide → Archive → Calibrate. Automated scanning of configured sources.
- (capability) CAP-1: Web dashboard accessible via domain name from any browser. Single-user personal deployment on VPS.
- (capability) CAP-2: Opportunity radar — prioritized list of buy/sell opportunities from all configured sources, sorted by net margin score, with swipe actions (treat/ignore) on mobile and table view on desktop.
- (capability) CAP-3: Opportunity detail — full margin breakdown (purchase price, shipping, platform commission, transaction fees), confidence score justification, sparkline price history, source link.
- (capability) CAP-4: Strategy configuration — adjustable thresholds (min margin, min ROI, min confidence, max risk), scoring weights, notification preferences, saved presets (Standard/Aggressive/Conservative/Custom).
- (capability) CAP-5: Source management — add/configure/remove data sources (Cardmarket, eBay, manual entry), per-source health monitoring (OK/degraded/blocked), manual scan trigger, error journal.
- (capability) CAP-6: Automated collection — periodic scanning of configured sources for card availability and pricing, normalized across sources, with configurable frequency.
- (capability) CAP-7: Net margin scoring — configurable scoring engine combining rentability (net margin, ROI), urgency (freshness, price trend), and confidence (source count, data quality) into a single actionable score.
- (capability) CAP-8: Telegram notifications — critical alerts pushed to Telegram when an opportunity exceeds configured thresholds, with summary data and direct link to radar.
- (capability) CAP-9: Decision history — per-opportunity action logging (treated/ignored/watched), stats dashboard for weekly calibration, missed opportunities analysis.
- (capability) CAP-10: Manual opportunity entry — inject an opportunity manually (product, purchase price, fees, estimated resale) when sources are unavailable or for niche finds.
- (capability) CAP-11: Responsive layout — mobile-first design from 320px to 1920px+. Mobile: 1-column cards, bottom tab bar (Radar/Detail/Settings/Sources). Desktop >=1024px: split-view Radar+Detail, side navigation, dense table.
- (capability) CAP-12: Product referential — managed catalog of tracked products (cards, sets, series) with auto-completion for manual entry, import/export capabilities.
- (constraint) Single-user deployment on VPS. Not designed for multi-tenancy or concurrent users.
- (constraint) Rust backend + React frontend. Stack is fixed — no framework migration in scope.
- (constraint) SQLite database. No external DB server dependency.
- (constraint) French UI language mandatory. i18n architecture present but no other locale implemented yet.
- (constraint) Touch targets >=44px on mobile. AA contrast minimum, AAA on KPIs. Keyboard navigable.
- (constraint) Deployed via reverse proxy (nginx/caddy) for HTTPS termination. No built-in TLS.
- (constraint) Amber Warmth light theme (#FFFDF7 bg, #D97706 accent, #2D2412 text). Dark mode deferred post-MVP.
- (decision) Non-goal: Multi-tenancy, SaaS, user accounts, authentication UI. Single-user only.
- (decision) Non-goal: Dark mode, push notifications, PWA, native mobile apps. All deferred post-MVP.
- (decision) Non-goal: Multiple languages. French only for MVP with i18n architecture ready.
- (decision) Non-goal: Automated purchasing, payment integration, inventory management. Signal-to-decision only.
- (decision) Success signal: Loris opens Poke-Radar from his iPhone, sees a Cardmarket alert for a tracked card with net margin > threshold, swipes to treat it, and the decision is logged — all within 5 minutes of the alert firing.
