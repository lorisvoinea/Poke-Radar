- (decision) AD-001 — Stack applicatif web. Remplace l'ancien ADR-001 Tauri desktop. Backend Rust avec Axum exposant une API RPC pragmatique (POST /api/*, sérialisation serde), frontend React/TypeScript SPA. Tout le domaine métier reste en Rust. La SPA est servie comme ressource statique. Paradigme: RPC léger plutôt que REST complet — chaque endpoint est une fonction, pas une ressource CRUD.
- (decision) AD-002 — Déploiement et infrastructure. Le backend Rust tourne comme un daemon systemd avec restart=always. Caddy agit en reverse proxy devant le backend, termine le TLS via Let's Encrypt automatique, et route toutes les requêtes. Les logs applicatifs passent par journald (via tracing-subscriber + journald layer). Build reproductible: une commande cargo build --release produit le binaire. Les secrets (token Telegram, clé auth, API keys) sont injectés via variables d'environnement, jamais dans le dépôt. Les migrations SQLite sont versionnées et appliquées au démarrage, garantissant la mise à jour sans perte de données.
- (decision) AD-003 — Authentification single-user par token statique. Un secret partagé (POKE_RADAR_AUTH_TOKEN en variable d'environnement) est comparé côté backend. Flux: page de login minimaliste dans la SPA → saisie token → POST /api/auth/verify → si match, stockage localStorage + header Authorization: Bearer à chaque appel. Pas de cookie, pas de session, pas de table users. Le token ne transite jamais dans l'URL.
- (decision) AD-004 — Architecture pipeline modulaire (ingestion → normalisation → scoring → alerting). Étapes isolées et testables indépendamment. Interfaces de données stables entre étapes. Fallback possible étape par étape. [ADOPTED — repris de l'ADR-002 de l'archi 2026-02, confirmé valide par Loris le 2026-07-22]
- (decision) AD-005 — Persistance locale SQLite + migrations versionnées. Zéro dépendance infra. Indexation ciblée pour requêtes dashboard et déduplication d'alertes. Timestamps UTC ISO-8601, montants en centimes (i64), audit de calcul pour explicabilité. [ADOPTED — repris de l'ADR-003 de l'archi 2026-02]
- (decision) AD-006 — Connecteurs par source avec stratégie de résilience unifiée. Chaque source implémente un trait Connector (collecte, mapping, health, erreurs typées). Retry/backoff/jitter et règles anti-abus mutualisées dans le runtime. Suivi de santé par source. [ADOPTED — repris de l'ADR-004 de l'archi 2026-02]
- (decision) AD-007 — Moteur de marge nette centralisé en Rust. Calcul économique unique (frais, commissions, port, hypothèses). Détail de calcul persisté avec chaque opportunité pour explicabilité. [ADOPTED — repris de l'ADR-005 de l'archi 2026-02]
- (decision) AD-008 — Alerting asynchrone avec anti-duplication. File interne + worker dédié, découplé du pipeline principal. Délai API Telegram ne bloque pas la détection. Gestion retry/backoff par canal. [ADOPTED — repris de l'ADR-006 de l'archi 2026-02]
- (decision) AD-009 — Scheduler intégré au backend Rust. Au démarrage, le backend lance une tâche tokio::spawn avec un interval configurable (tokio::time::interval) qui déclenche les cycles de collecte automatiquement. Jitter et quotas par source gérés dans le runtime. Si le process tombe, systemd le relance et le scheduler repart. Zéro dépendance externe.
- (decision) AD-010 — Communication temps réel frontend↔backend via Server-Sent Events (SSE). Le backend expose un endpoint GET /api/events qui pousse les événements (nouvelle opportunité, alerte envoyée, changement santé source) au frontend. Le frontend utilise EventSource natif. Pas de WebSocket (pas de besoin bidirectionnel). Complété par les appels RPC POST /api/* pour les actions utilisateur.
- (decision) AD-011 — Environnement unique (VPS = prod). Pas de staging/dev séparé. Configuration via variables d'environnement injectées par systemd (EnvironmentFile ou Environment). Une seule source de vérité pour la config, pas de profils.
- (decision) AD-012 — Structure projet: workspace classique. Racine avec backend/ (Cargo, Axum, domaine métier, connecteurs) et frontend/ (Vite + React/TypeScript SPA). Le backend inclut le sous-dossier frontend/dist/ comme ressource statique servie par Axum en production.
- (decision) AD-013 — Tests unitaires Rust. Domaine métier (marge, scoring, normalisation, déduplication) couvert par des tests #[cfg(test)] dans le même fichier. Chaque règle de calcul doit avoir au moins un cas nominal et un cas outlier. Les connecteurs sont testés avec des fixtures HTTP mockées (wiremock ou httpmock).
- (decision) AD-014 — Tests d'intégration Rust. Tests dans le répertoire tests/ du crate backend, exécutant le pipeline complet (ingestion → normalisation → scoring → alerting) sur des données de test. Base SQLite en mémoire (:memory:) pour l'isolation. Les connecteurs sont mockés, le reste du pipeline est réel.
- (decision) AD-015 — Tests frontend React. Vitest + React Testing Library pour les tests unitaires de composants. Chaque composant partagé (Badge, Button, Card, Toast, etc.) est testé pour ses états (default, disabled, loading, error). Les pages (RadarPage, DetailPage, StrategyPage, SourcesPage) sont testées avec un mock du backend (MSW — Mock Service Worker).
- (decision) AD-016 — Tests end-to-end. Playwright pour les tests du flux complet utilisateur. Scénarios clés: journey 1 (alerte → décision en <5min), journey 2 (recalibrage hebdomadaire), journey 3 (source indisponible → fallback manuel), journey 4 (saisie manuelle d'opportunité). Le backend réel est lancé avant les tests (pas de mock). La base SQLite de test est recréée à chaque run.
- (decision) AD-017 — Sécurité web. CORS restreint au domaine configuré (pas de wildcard). Rate limiting par IP sur les endpoints sensibles (auth, scan manuel). Requêtes SQL paramétrées exclusivement (pas de concatenation). Header Content-Security-Policy restrictif. Protection CSRF non nécessaire (pas de cookies, token en header Authorization). HTTPS obligatoire (via Caddy).
- (decision) AD-018 — Conventions de nommage et formats. Rust: snake_case. TypeScript: camelCase. SQL: snake_case explicite. Erreurs API: format uniforme { error: string, code: string }. Montants en centimes (i64). Timestamps UTC ISO-8601. Correlation_id par cycle et par opportunité pour observabilité. [ADOPTED]
- (version) [ASSUMPTION] Stack versions — web search indisponible. Rust stable (latest), Axum 0.8, React 19, TypeScript 5.8, Vite 6, SQLite via rusqlite 0.32, Vitest 3, Playwright 1.52, React Testing Library 16. À vérifier avant implémentation.
